# Requirements Document

## Introduction

This document defines the requirements for the Forgot Password feature in the Finance AI app. The feature enables manual (non-Google) users to securely reset their password via a time-limited, single-use email token. The flow is entirely inline on the login page for the request step, and uses a dedicated `reset-password.html` page for the actual password change. Google OAuth users are explicitly excluded from this flow.

## Glossary

- **System**: The Finance AI application (frontend + backend combined)
- **Server**: The Node.js/Express backend (`server.js`)
- **Login_Page**: The `login.html` frontend page
- **Reset_Page**: The `reset-password.html` frontend page
- **Forgot_Section**: The inline `#forgot-section` div rendered within `login.html`
- **User**: A registered account stored in MongoDB
- **Manual_User**: A User whose account was created with email/password (no `googleId` field set)
- **Google_User**: A User whose account was created via Google OAuth (`googleId` field is set)
- **Reset_Token**: A cryptographically secure 64-character hex string generated by `crypto.randomBytes(32)`
- **Token_Expiry**: A timestamp set to exactly 1 hour after token generation
- **Nodemailer**: The already-configured Gmail SMTP email transporter in `server.js`

## Requirements

### Requirement 1: Inline Forgot Password Trigger

**User Story:** As a manual user on the login page, I want to click "Forgot password?" and see a password reset form inline, so that I can initiate a reset without leaving the page.

#### Acceptance Criteria

1. WHEN a user clicks the "Forgot password?" link on the Login_Page, THE Login_Page SHALL hide the login form and display the Forgot_Section inline without any page navigation.
2. WHEN the Forgot_Section is displayed, THE Login_Page SHALL focus the email input field automatically.
3. WHEN a user clicks the "Back to login" link within the Forgot_Section, THE Login_Page SHALL hide the Forgot_Section and restore the login form to its original visible state.

---

### Requirement 2: Forgot Password Email Submission

**User Story:** As a manual user, I want to submit my email address to receive a password reset link, so that I can regain access to my account.

#### Acceptance Criteria

1. WHEN a user submits the Forgot_Section form with a valid email address, THE Login_Page SHALL send a POST request to `/api/forgot-password` with the email in the request body.
2. WHEN the forgot-password request is in progress, THE Login_Page SHALL disable the submit button and display a loading indicator.
3. WHEN the Server returns a success response, THE Login_Page SHALL display an inline success message instructing the user to check their inbox.
4. WHEN the Server returns an error response, THE Login_Page SHALL display the error message inline within the Forgot_Section without redirecting.
5. IF the email input field is empty when the form is submitted, THEN THE Login_Page SHALL prevent form submission and display a validation error.

---

### Requirement 3: Forgot Password API — Email Enumeration Prevention

**User Story:** As a security-conscious system operator, I want the forgot-password endpoint to never reveal whether an email address is registered, so that attackers cannot enumerate valid accounts.

#### Acceptance Criteria

1. WHEN a POST request is made to `/api/forgot-password` with an email that does not match any User document, THE Server SHALL return HTTP 200 with the message `"If that email exists, a reset link has been sent."` and perform no database mutation.
2. WHEN a POST request is made to `/api/forgot-password` with any email, THE Server SHALL never return HTTP 404 or any response that distinguishes between a registered and unregistered email.
3. IF the `email` field in the request body is empty or missing, THEN THE Server SHALL return HTTP 400 with the message `"Email is required."`.

---

### Requirement 4: Forgot Password API — Google User Guard

**User Story:** As a Google OAuth user, I want to be clearly informed that I cannot use the password reset flow, so that I am directed to the correct sign-in method.

#### Acceptance Criteria

1. WHEN a POST request is made to `/api/forgot-password` with an email belonging to a Google_User (a User document where `googleId` is set), THE Server SHALL return HTTP 400 with the message `"This account uses Google Sign-In. Please sign in with Google."`.
2. WHEN a POST request is made to `/api/forgot-password` with an email belonging to a Google_User, THE Server SHALL not generate a Reset_Token or write any token fields to the User document.

---

### Requirement 5: Forgot Password API — Token Generation and Email Dispatch

**User Story:** As a manual user, I want to receive a password reset email with a secure link, so that I can navigate to the reset page and set a new password.

#### Acceptance Criteria

1. WHEN a POST request is made to `/api/forgot-password` with an email belonging to a Manual_User, THE Server SHALL generate a Reset_Token using `crypto.randomBytes(32).toString('hex')`.
2. WHEN a Reset_Token is generated, THE Server SHALL set `user.resetToken` to the generated token and `user.resetTokenExpiry` to `Date.now() + 3600000` (1 hour from now) and persist both fields to the database.
3. WHEN the token is saved, THE Server SHALL send a reset email via Nodemailer to the user's email address containing a reset link of the form `{FRONTEND_URL}/reset-password.html?token={token}`.
4. IF Nodemailer fails to send the reset email, THEN THE Server SHALL return HTTP 500 with the message `"Failed to send reset email. Please try again."` and SHALL NOT persist the Reset_Token to the database.
5. WHEN the reset email is sent successfully, THE Server SHALL return HTTP 200 with the message `"If that email exists, a reset link has been sent."`.

---

### Requirement 6: Reset Password Page — Token Extraction and Validation

**User Story:** As a user who clicked a reset link in my email, I want the reset page to read my token from the URL and show me the password form, so that I can set a new password.

#### Acceptance Criteria

1. WHEN the Reset_Page loads, THE Reset_Page SHALL extract the `token` query parameter from the URL using `URLSearchParams`.
2. IF the `token` query parameter is absent or empty when the Reset_Page loads, THEN THE Reset_Page SHALL display an error message `"Invalid reset link. Please request a new one."` and hide the password reset form.
3. WHILE a valid token is present in the URL, THE Reset_Page SHALL display the new password and confirm password input fields.

---

### Requirement 7: Reset Password Page — Client-Side Validation

**User Story:** As a user setting a new password, I want immediate feedback if my inputs are invalid, so that I can correct them before submitting.

#### Acceptance Criteria

1. WHEN a user submits the reset form with a new password and confirm password that do not match, THE Reset_Page SHALL display the error message `"Passwords do not match."` and not submit the form to the Server.
2. WHEN a user submits the reset form with a new password shorter than 6 characters, THE Reset_Page SHALL display the error message `"Password must be at least 6 characters."` and not submit the form to the Server.

---

### Requirement 8: Reset Password API — Token Validation

**User Story:** As a user, I want the system to reject expired or invalid reset tokens, so that my account is protected from unauthorized password changes.

#### Acceptance Criteria

1. WHEN a POST request is made to `/api/reset-password` with a token that does not match any User document's `resetToken` field, THE Server SHALL return HTTP 400 with the message `"Reset link is invalid or has expired."` and perform no database mutation.
2. WHEN a POST request is made to `/api/reset-password` with a token whose corresponding `resetTokenExpiry` is less than or equal to the current time, THE Server SHALL return HTTP 400 with the message `"Reset link is invalid or has expired."` and perform no database mutation.
3. IF the `token` or `newPassword` field is empty or missing in the request body, THEN THE Server SHALL return HTTP 400 with the message `"Token and new password are required."`.

---

### Requirement 9: Reset Password API — Password Update and Token Invalidation

**User Story:** As a user with a valid reset token, I want my new password to be saved and the token to be invalidated immediately, so that the reset link cannot be reused.

#### Acceptance Criteria

1. WHEN a POST request is made to `/api/reset-password` with a valid, non-expired token, THE Server SHALL hash the new password using `bcrypt.hash(newPassword, 10)` and save it to `user.password`.
2. WHEN the password is updated successfully, THE Server SHALL set `user.resetToken` to `null` and `user.resetTokenExpiry` to `null` and persist these changes to the database.
3. WHEN the password is updated successfully, THE Server SHALL return HTTP 200 with the message `"Password reset successful."`.
4. WHEN a Reset_Token has been used in a successful password reset, THE Server SHALL reject any subsequent POST request to `/api/reset-password` using the same token with HTTP 400.

---

### Requirement 10: Reset Password Page — Post-Reset Redirect

**User Story:** As a user who has successfully reset their password, I want to be redirected to the login page, so that I can sign in with my new credentials.

#### Acceptance Criteria

1. WHEN the Server returns a success response to a reset-password submission, THE Reset_Page SHALL display the message `"Password reset! Redirecting to login..."`.
2. WHEN the success message is displayed, THE Reset_Page SHALL redirect the user to `login.html` after a short delay.
3. WHEN the Server returns an error response to a reset-password submission, THE Reset_Page SHALL display the error message inline and allow the user to request a new reset link.

---

### Requirement 11: User Model — Reset Token Fields

**User Story:** As a developer, I want the User model to store reset token data, so that the backend can validate and manage password reset requests.

#### Acceptance Criteria

1. THE System SHALL extend the User Mongoose schema with a `resetToken` field of type `String` with a default value of `null`.
2. THE System SHALL extend the User Mongoose schema with a `resetTokenExpiry` field of type `Date` with a default value of `null`.
3. WHEN a password reset is completed successfully, THE System SHALL ensure `resetToken` and `resetTokenExpiry` are both `null` on the User document.
4. WHEN a Google_User document is saved, THE System SHALL never have `resetToken` set to a non-null value.
